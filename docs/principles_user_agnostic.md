# 이 시스템을 만들 때 지킬 약속들

이 문서는 시스템을 짜다가 헷갈릴 때 펴서 보는 가이드야.

우리는 이 시스템을 **나(코비) 혼자만 쓸 게 아니라 다른 사람도 딸깍 누르면 자기 보고서가 나오는 형태**로 만들기로 했어. 근데 내 데이터만 보면서 짜다 보면 자연스럽게 내 작업 방식에 딱 맞춘 시스템이 돼버려. 그러면 다른 사람한테 줘도 결과가 이상하게 나옴.

그걸 막기 위해 5개 약속을 미리 박아두는 거야.

---

## 약속 1. 내 데이터는 "검증용"으로만 쓰고, "튜닝용"으로는 안 쓴다

**무슨 말이냐면:**

시스템을 만들고 나서 내 데이터에 돌려봤을 때 결과가 마음에 안 들 수 있어. 그때 두 가지 반응이 가능해:

- 반응 A (튜닝): "결과가 이상하네. 내 데이터에 맞게 룰을 고치자"
- 반응 B (검증): "결과가 이상하네. 우리가 세운 가정 중에 뭐가 틀렸지?"

A로 가면 내 데이터에 딱 맞는 시스템이 됨. B로 가면 일반화된 시스템이 유지됨.

**언제 헷갈리느냐:**

A와 B의 경계가 미묘해. 예를 들어 흐림률이 12% 안 나오고 18%가 나왔다고 치자. 그때:

- ❌ "Read 도구 carve-out을 80줄 → 120줄로 늘리자" (= A, 내 Read 사용 패턴에 맞춤)
- ⭕ "왜 18%지? 우리가 '문서형 출력'이라고 분류한 도구들이 실제론 더 다양한가 보다. 출력 유형 분류 기준을 다시 보자" (= B)

**자주 마주치는 함정 — yaml 룰 깎기**

신호 정의를 yaml 로 빼두면 안전한 것처럼 느껴지지만, 본인 데이터에 안 보이는 항목을 빼는 것도 약속 1 위반이야.

- ❌ "내 데이터에 `git bisect` 가 0회네. 명세에서 빼자" → 다른 사람은 bisect 를 자주 쓸 수 있음. **유지.**
- ❌ "내 데이터엔 `pr_with_structured_body` 가 안 잡히네. 정규식 더 느슨하게" → 본인 PR 스타일에 끼워 맞추기. **유지.**
- ⭕ "내 데이터에 outcome 이 하나도 안 잡히는 에피소드가 60%네. 명세상 outcome 이 정의되는 조건을 다시 보자 → `episode_kind` 분류가 빠져있었구나" → 가정 점검 후 일반화된 보완.

차이의 핵심: **내 데이터에 맞춰 룰을 좁히거나 늘리는 건 위반. 명세에 빠진 일반 개념을 발견해 추가하는 건 OK.**

**예외:**

진짜 버그(파싱 에러, 정규식 오작동)는 고침. "결과가 기대와 다르다"는 안 고침.

## 약속 2. 사람마다 다른 정보는 코드에 박지 않는다

**무슨 말이냐면:**

코드 안에 `if tool_name == "grafana_query_logs"` 이런 게 박혀 있으면, 이건 "내가 grafana 쓴다"를 가정한 거야. 다른 사람은 datadog이나 splunk 쓸 수 있음.

해결책: **도구 이름 자체가 아니라 "그 도구가 어떤 기능인가"를 코드에 박는다.**

- 로그 검색 기능 → 사람마다 grafana / datadog / splunk 등 다름
- trace 조회 기능 → 사람마다 datadog / honeycomb 등 다름
- 이슈 트래커 기능 → 사람마다 JIRA / Linear 등 다름

이런 식으로 **기능 그룹**으로 묶고, "어느 도구가 어느 기능에 속하는지"는 사용자별로 자동 매핑하면 됨. 기능 그룹은 코드에 박아도 OK (대부분 기능별로 1-2개의 도구만 존재하니까).

**박으면 안 되는 것들 / 박아도 되는 것들:**

| 정보 | 처리 방법 |
|---|---|
| 도구 → 기능 그룹 매핑 | 기능 그룹은 코드에 박음. 어느 도구가 어느 그룹인지는 자동 분류 + 사용자 확인 |
| 기능 그룹별 출력 처리 룰 | 코드에 박음 (예: "log_search 기능은 첫 3개 로그만 보관") |
| 카테고리 (prod_incident, refactoring 등) | LLM이 자유 텍스트로 답함. enum 없음 |
| 회사명/내부 시스템명 (WMS, OMS, kakaostyle) | 사용자가 처음 한 번 "마스킹할 단어" 입력 |
| 자주 쓰는 워크플로 | 결과물에서 발견. 미리 정해두지 않음 |

**기능 그룹 초안 (확장 가능):**

```
log_search       — grafana, datadog logs, splunk, loki, ...
trace_view       — datadog APM, honeycomb, jaeger, ...
issue_tracker    — JIRA, Linear, GitHub Issues, ...
code_search      — Grep, ripgrep, GitHub code search, ...
file_read        — Read (Claude Code 기본), ...
file_edit        — Edit, Write, ...
shell_exec       — Bash, ...
web             — WebFetch, WebSearch, ...
chat            — Slack, Discord MCP, ...
docs            — Confluence, Notion MCP, ...
```

새 기능이 필요해지면 그룹 추가. 예: "monitoring_alert", "feature_flag", "design_tool" (Figma) 등.

**예시:**

❌ 안 되는 코드:
```python
TOOL_CARVE_RULES = {
    "grafana_query_logs": "first_3_logs",      # 사용자가 splunk면 안 맞음
    "atlassian_jira_search": "title_only",     # 사용자가 Linear면 안 맞음
}
```

⭕ 되는 코드:
```python
# 기능 그룹별 룰 (코드에 박힘 OK)
CARVE_RULES_BY_FUNCTION = {
    "log_search": "first_3_logs",
    "trace_view": "first_1_trace",
    "issue_tracker": "title_and_status",
    "file_read": "signature_or_full",
    # ...
}

# 사용자별 도구 → 기능 매핑 (자동 분류 + 사용자 1회 확인 후 캐시)
user_tool_map = {
    "grafana_query_logs": "log_search",
    "datadog_search_spans": "trace_view",
    "atlassian_jira_search": "issue_tracker",
    # ...
}
```

**그래도 출력 유형 분류가 필요한 경우:**

Bash, Read 같은 **기본 도구**는 용도가 다양해서 기능 그룹 하나로 못 묶음. 
- Bash는 grep도 쓰고 build도 쓰고 test도 씀 → 출력이 매번 다름
- Read는 코드 파일도 보고 로그 파일도 보고 마크다운도 봄 → 출력이 매번 다름

이건 출력 유형(리스트형/실행형/문서형) 분류가 필요. 단 Bash/Read 자체는 Claude Code 표준 도구라 박아도 OK.

## 약속 3. 다른 사람이 검증을 직접 해야 한다면 그건 실패다

**무슨 말이냐면:**

지금까지 검증 1-3에서 내가 한 일들:
- 30개 에피소드 손으로 라벨링 (검증 2)
- 보고서 Q1/Q2/Q3 항목별 채점 (검증 3)
- 흐림률 32% → 12% 손으로 계산 (검증 1)

이거 다른 사람은 절대 안 함. 다른 사람은 그냥 결과 받고 싶어해.

**그래서:**

- 검증 1-3은 **내가 baseline 세우는 용도로 1회만 함**. 다른 사람은 안 함
- 다른 사람용 신호는 가벼운 것만:
  - 보고서 끝에 "1-5 별점 매기기" (옵션, 안 해도 됨)
  - 베타 단계에서 비공식 피드백 받기

내 검증 작업은 **다른 사람들을 대신해서 한 번 빡세게 해보고 "이 정도면 자동 결과 믿어도 된다"는 기준선을 만드는 일**이야. 매번 반복하는 게 아니라.

**약속 3 재해석 — 출력 모드 운영 모델**

이 시스템은 두 가지 출력 모드를 가진다:

- **디폴트 (광역 모드)**: 사소한 패턴까지 다 보고서에 포함. **사용자가 1차 검수자 역할** 을 맡아 첨삭 후 동료에게 공유함. 사용자에게 "라벨링" 같은 손작업을 강제하지 않으므로 약속 3 위반이 아님 (검증이 아니라 편집 — 자기 보고서의 톤을 가다듬는 것)
- **`--curated` 모드**: 5-8 패턴만 추출, 1회성 폐기. 사용자가 첨삭 없이 곧장 동료에게 공유할 수 있는 형태. 약속 3 의 "다른 사람은 그냥 결과 받고 싶어해" 에 부합.

→ 약속 3 의 핵심은 **"검증 손작업을 강제하지 않는다"** 임. 광역 모드는 사용자 첨삭을 가정하지만, 그건 본인 책임 영역의 편집이지 검증 손작업 강제가 아님.

→ 다른 사람에게 시스템을 줄 때는 `--curated` 디폴트를 권고할지 사용자가 그때 판단하면 됨. 본인용 디폴트와 동료용 디폴트가 다를 수 있다는 게 이 시스템의 운영 모델.

## 약속 4. 최종 보고서는 사내 동료가 읽고 따라할 수 있어야 한다

**무슨 말이냐면:**

이 시스템의 보고서는 **사내 동료끼리만 공유**한다. 외부 공유는 안 함.

그래도 두 가지는 지켜야 함:
1. 어떤 도구를 썼는지 알 수 있어야 함 (사내 동료라도 모든 MCP를 다 알진 못함)
2. 새지 말아야 할 것은 새지 말아야 함 (시크릿, 의도치 않은 민감 정보)

회사 내부 시스템명(WMS, OMS, kakaostyle 등)은 동료들이 다 아니까 **그대로 노출**. 추상화하면 오히려 모호해짐.

**처리 방식:**

| 종류 | 처리 |
|---|---|
| **시크릿** (API 키, 토큰, 비밀번호 등) | 정규식으로 자동 마스킹. 항상 ON. 끄기 불가. |
| **동료/사람 이름** | 디폴트 ON (관찰자 효과 방지). 사용자가 끌 수 있음. |
| **회사 시스템명** (WMS, OMS 등) | 마스킹 OFF. 그대로 노출. |
| **일반 공개 도구** (grafana, datadog 등) | 그대로 노출 + 첫 등장 시 인라인 괄호로 짧은 기능 설명. |

**도구 인라인 괄호 표기:**

사내 동료라도 모든 MCP를 다 알진 못하니까 처음 등장 시 괄호로 설명.

- 첫 등장: `grafana mcp(로그 검색 도구)`
- 두 번째 등장부터: `grafana mcp` 만

**예시:**

❌ 안 좋은 표현 (도구 모르는 사람 못 따라함):
> "동시성 충돌 발생 시 로그 검색 도구에서 관련 로그를 일괄 수집"

⭕ 좋은 표현:
> "동시성 충돌 발생 시 **grafana mcp(로그 검색 도구)**에서 관련 로그를 일괄 수집한 뒤 WMS의 인벤토리 락 추적 시작"

WMS, OMS 같은 사내 시스템명은 그대로 들어감. 동료들이 봐야 더 구체적으로 도움됨.

**관찰자 효과 방지 (동료 이름 마스킹):**

패턴에 "[동료 이름]이 검토 요청" 같은 게 그대로 나오면 그 동료가 본인 의지와 무관하게 노출됨. 디폴트로 동료 이름은 "동료" 또는 역할명으로 추상화. 사용자가 옵션으로 끌 수 있음.

**수동으로 차단 (옵션):**
- 보고서가 공유되기 전에 본인이 한 번 보고 검토 가능

## 약속 5. 헷갈릴 때마다 이 질문 4개에 답해본다

작업 중 룰을 만들거나 고칠 때 잠깐 멈추고 물어보기:

**질문 1. 이 룰을 다른 사람한테도 똑같이 적용 가능한가?**
- "당연하지" → OK
- "음... 사람마다 다를 수도" → 일반화 부족. 다시 작업

**질문 2. 이 룰을 짠 출발점이 뭐였나?**
- "일반 원리에서 시작했다" → OK
- "내 데이터에서 이런 패턴이 보여서" → 위험. 더 추상화 가능한지 검토

**질문 3. 이 룰을 코드 밖으로 (설정 파일이나 데이터로) 뺄 수 있나?**
- 뺄 수 있으면 지금 뺀다. 지금 안 빼면 영영 못 뺌
- 못 빼면 왜 못 빼는지 명확히 (진짜 못 빼는 건 거의 없음)

**질문 4. 이 시스템을 프론트엔드 개발자나 디자이너에게 줘도 돌아가나?**
- 그 사람은 Bash(grep) 안 쓰고, Figma MCP 쓸 수도 있고, 카테고리도 완전 다름
- file_edit 자체가 거의 없는 직군(PM/QA) 도 있음 — 이 경우 에피소드 다수가 `investigation_only` 로 분류되어 종료부 패턴이 발현 안 됨. 그래도 도입부 패턴과 시간축 진단 패턴은 잡힘
- 그래도 의미 있는 결과 나오면 OK
- 안 나오면 일반화 실패

## 시작하기 전 사전 합의 사항

이 약속들을 지키면 시스템이 일반화되는 대신 **내 데이터 결과 품질은 살짝 떨어질 수 있어**.

지금까지 본인 최적화 결과가 3.85점이었다면, 일반화 후엔 3.5-3.7점 정도가 나올 수도 있음. 대신 다른 사람한테 줘도 비슷한 품질이 나옴.

이 트레이드오프 받아들이고 시작함.
