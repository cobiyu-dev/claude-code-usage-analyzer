# Format C — 보고서 형식 가이드

Claude(자신)가 Stage D 작업 중 보고서 작성할 때 참조하는 형식 명세.

배경/결정 사유는 `docs/format_c_spec.md` 참고. 이 문서는 **"이렇게 작성하라"** 지시만 포함.

---

## 보고서 전체 구조

```markdown

[메타 헤더]

---

[패턴 — 느슨한 그룹화 또는 평탄]

---

[주차별 또는 월별 요약]

---

## 사용 통계

[자주 쓴 기능/도구 top N]

---

## 내 로컬 셋업

[다른 사람과 공유 안 하는 ~/.claude/ 의 개인 설정 — 자연어 설명 톤. aggregated.json 의 local_setup 키 기반. 자세한 형식은 docs/format_c_spec.md "2.5 개인 로컬 셋업 섹션" 참조]

[원본 데이터 위치, 오류 신고 등]
```

섹션 제목 규칙:
- "주요 워크플로 패턴" → **"자주 보이는 작업 방식"** (워크플로 같은 외래어 안 씀)
- "시계열 추이" → **"주별 활동 추이"** (월 단위면 "월별 활동 추이")
- "사용 메타 정보" → **"사용 통계"**
- "개인 로컬 셋업" → **"내 로컬 셋업"** ("개인" 같은 격식 어휘 빼기)
- "raw 데이터" → **"원본 데이터"**

```markdown
**대화 세션 수**: 47개
```

라벨 어휘 규칙:

## 2. 패턴 한 개 형식

```markdown
#### [그룹문자]-[번호]. [짧은 라벨 — 2-5단어]

> [한 줄 제목 — 상황 + 행동을 한 줄로 요약]

[1-2 문장 짧은 본문. 패턴의 핵심을 풀어 씀.]

- **언제**: [이 패턴이 발동되는 상황·트리거 — 한 줄]
- **어떻게**: [구체적 행동·발화 모양 — 한 줄. 사용자 발화 예시를 따옴표로 짧게]
- **왜**: [이 패턴이 가져다 주는 효과·동기 — 한 줄]

**사용 예시 ([도메인 라벨])**

상황: [한 줄로 어떤 상황인지]
이렇게 풀어갔다 ↓

1. [의도 한 줄]
   → "[실제로 친 메시지]"

2. [AI 응답 + 다음 의도]
   → "[실제로 친 메시지]"

3. ...

---
```

### 핵심 규칙

- **헤더는 `#### [그룹문자]-[번호]. [짧은 라벨]` 형식** (예: `A-1. 사실 수집 위임`)
- **한 줄 제목은 인용구** (`> 한 줄`) 로 패턴 본질을 한 줄로
- **본문은 1-2 문장**. 길게 한 호흡으로 풀지 말 것
- **언제/어떻게/왜 3줄 bullet 은 모든 패턴에 필수** — 본문을 길게 풀어 쓰지 말고 bullet 에서 쪼갤 것
- 빈도/통계 표시 X (예: "12회 관찰" 같은 거 X)
- 도구 이름은 일반 공개 도구만 표기. 회사 내부 시스템명은 본문에 그대로 노출 가능 (사내 공유)
- 회사 내부 메서드명/엔티티명은 추상화 (`SubmitFCWorkerPackingOutbound` → `특정 워커` 등)
- **사용 예시 블록은 모든 패턴에 필수**. 본문 직후 코드 블록 형태 (단계 1-2-3-4)

### 가독성 룰 — 다른 직군 개발자 기준

받는 사람은 coding agent 를 쓰지만 시스템 내부 구조는 모른다. 다음 두 종류 어휘 금지:

**금지 1: 시스템 내부 라벨**
`outcome / phase / intro / main / verify / git_intent / diagnostic / episode_kind / with_changes / investigation_only / tooling_only / mini_pattern / situation_cluster / function_group / committed / pushed / pr_opened / verified_by_data / verified_by_run / delegated_and_reported / incremental_commits / single_final_commit / pr_with_structured_body / abandoned_or_paused`

본문·시계열·메타 정보 어디에도 등장 X.

**금지 2: Claude Code 도구 내부명**
`Bash, Edit, Read, Write, Grep, Glob, Task, TaskCreate, TaskUpdate, ToolSearch, ExitPlanMode, AskUserQuestion, Agent`

본문에선 일반 동사로:
- Bash → 셸 실행 / 명령 실행
- Edit → 파일 수정
- Read → 파일 읽기
- Grep/Glob → 코드 검색, 파일 경로 검색
- Task/Agent → **별도 context 서브 agent에게 위임** (지금 창 안에서 깨끗한 컨텍스트로 띄우는 보조 AI. "별도 창" 이 아님 — 주의)
- TaskCreate/TaskUpdate → 단계별 to-do 관리
- ToolSearch → 필요한 도구 찾아 로딩
- ExitPlanMode → 계획 모드(Plan Mode)의 계획 확정

**용어 구분 (중요 — 헷갈리지 말 것)**
- `Task/Agent` 도구 = **별도 context 서브 agent** = 지금 작업 중인 창 안에서 깨끗한 컨텍스트를 가진 보조 AI를 띄워 작업을 맡기는 것. 사용자가 평소 "별도 context agent에게 시켜줘" 식으로 가장 많이 쓰는 방식. → 보고서엔 **"별도 context 서브 agent"** 로 표기.
- `teammate` (tmux 별도 세션) = 화면상 **진짜 다른 창**에서 독립적으로 도는 동료 AI. 사용자는 이건 거의 안 씀. → 굳이 등장하면 "별도 창 동료 AI" 로 표기. **둘을 섞어 쓰지 말 것.**

**유지 OK**
- AI agent 추상 구조: 별도 context 서브 agent, Subagent, 계획 모드(Plan Mode), 슬래시 커맨드, /clear, fresh agent, cold review, ultrathink
- MCP 표기: `grafana mcp(로그 검색 도구)`, `mysql mcp(DB 쿼리 도구)` 그대로

### 사용 예시 블록 규칙

**핵심**: 보고서는 "어떤 패턴으로 coding agent 를 프롬프트하면서 사용하는가" 를 잡는 문서. **"어떤 명령어를 사용했는가"** 가 아님.

→ 사용 예시 한 단계 = **사용자가 친 메시지 한 번** (AI 가 호출한 도구·명령이 아님).

| ❌ AI 실행 시퀀스 | ✅ 사용자 프롬프트 시퀀스 |
|---|---|
| `grafana mcp 로 5xx 에러 조회` | `"운영에서 5xx 떨어지는 거 봐줘. 14시쯤"` |
| `git log -- src/auth.py 로 변경 확인` | `"이거 언제부터 이렇게 동작했지? 최근 변경 봐줘"` |
| `Edit 으로 cache TTL 조정` | `"이 부분 TTL 좀 줄여줘"` |

각 단계는 따옴표로 감싼 자연어 메시지. 짧은 승인("응 진행해줘", "그대로 해줘")도 OK.

도메인은 4가지 중 하나만 선택, 헤더에 명시 (`**사용 예시 (REST API 서버 디버깅):**`):
1. **REST API / 디버깅**
2. **인증 / 토큰**
3. **캐시 / DB**
4. **Git / 테스트 / 빌드**

회사 내부 시스템명·메서드명·사내 슬래시 스킬은 사용 예시 블록에 절대 들어가지 않음. 본문엔 OK, 예시엔 X.

### 사용 예시 블록 형식 (필수)

각 사용 예시는 **상황 한 줄 + 단계별 (의도 + 화살표 메시지) 두 줄 구성** 으로 작성한다:

```markdown
**사용 예시 ([도메인 라벨])**

상황: [한 줄로 어떤 상황인지 — 일상 표현]
이렇게 풀어갔다 ↓

1. [그 시점에 사용자가 뭘 하고 싶었는지 — 한 줄, 의도]
   → "[실제로 친 자연어 메시지]"

2. [AI 가 1번에 어떻게 응답했는지 + 다음에 뭐 하고 싶었는지]
   → "[실제로 친 메시지]"

3. ...
```

핵심:
- 각 단계 = **두 줄 구성** (윗줄: 사용자 의도, 아랫줄: → 화살표로 실제 친 메시지)
- AI 응답은 다음 단계 윗줄에 한 구절로 녹임 ("AI 가 ... 보여줘서, ...") — **별도 단계로 분리 X**
- "상황:" 한 줄로 작업 시작점 (본문과 중복되지 않게)
- 3-6 단계

### 사용 예시 예 (좋음)

```markdown
**사용 예시 (REST API 서버 디버깅)**

상황: 운영에서 user-api 가 갑자기 5xx 를 뱉기 시작.
이렇게 풀어갔다 ↓

1. 로그부터 끌어와 가설을 좁히고 싶다.
   → "운영에서 5xx 떨어지는 거 봐줘. 최근 1시간 user-api 쪽"

2. AI 가 에러 응답을 모아 보여줘서, 그중 응답 느린 사용자를 추리고 싶다.
   → "이 중에 응답 느린 user_id 몇 개만 뽑아줘"

3. 같은 호출 흐름을 인증 서비스에서도 이어서 보고 싶다.
   → "같은 trace_id 로 인증 쪽 로그도 이어서 봐줘"

4. 이제 코드 어디부터 봐야 할지 짚어 달라.
   → "여기까지 보니 어디 코드부터 들여다봐야 할 거 같아? 경로 짚어줘"
```

### 자가 점검 (사용 예시 작성 후)

- **각 단계가 두 줄 (의도 + → 메시지) 인가?**
- **상황 한 줄이 일상 표현으로 됐나?** (전문 용어 X)
- **AI 응답이 다음 단계에 자연스럽게 녹았나?** 별도 단계로 떨어져 있지 않은가
- 메시지가 사용자가 친 자연어 형태인가? AI 실행 시퀀스 형태 X
- 도메인 라벨이 4가지 중 하나인가?
- 회사 시스템명이 들어가지 않았나?
- 사내 슬래시/플러그인 이름이 들어가지 않았나?
- 메시지가 자연스러운가? 너무 격식 차린 명령조 X
- 단계 수가 3-6 사이인가?

### 본문 어조 (사용자 지시 어조 — 필수)

이 보고서는 "어떤 패턴으로 coding agent 를 프롬프트하는가" 를 잡는 문서.
본문 어조는 **사용자가 AI 에게 시키는 시점** 으로 통일한다.

- "AI 가 ... 한다" / 모호한 "... 한다" → "... 하도록 시킨다" / "... 하게 한다" / "... 게 시킨다"
- "... 직접 한다" (사용자 자기 손) → 보통 "... 직접 하게 시킨다"
  단, 사용자가 실제로 자기 손으로 하는 행동 (SQL 본문 작성, prod 에서 직접 쿼리 두드림 등) 은 그대로
- 어미는 자연스러운 한국어 범위 안에서만: "...게 한다 / ...도록 시킨다 / ...게 시킨다 / ...하게 만든다". 같은 어미가 3번 이상 연달아 나오지만 않게 하면 됨.
- **사용 금지 어미**: "...게 굴린다 / ...을 굴린다" — 한국 개발자가 일상에서 쓰지 않는 표현. 어미 다양화하려고 억지로 끼워 넣지 말 것.
- 길이 변화 최소. 어미만 바꾸거나 짧게 끊는다.
- **어조 변경 대상이 아닌 영역**:
  - 사용 예시 블록 안의 "의도" 줄·"→ 메시지" 줄 (이미 사용자 시점)
  - 시계열 추이·사용 메타·참고 섹션 (통계 톤)
  - 그룹 헤더 (충분히 추상화돼 있음)

### 본문 어휘 (가독성 — 한국 개발자 일상어 기준)

받는 사람은 **Claude Code 를 그냥 쓰기만 하는 한국 개발자**다. 분석가도, 학자도 아님. 본문은 사내 슬랙 메시지 정도의 일상 어휘로 풀어 쓴다.

**기본 원칙 1: 한자어·명사화 금지**

동사를 추상명사로 압축한 어휘는 쓰지 않는다. 풀어서 동사로 다시 쓴다.

| 금지 (한자어 / 명사화) | 풀어 쓰기 |
|---|---|
| 둘러싸기 / 둘러싸기 패턴 | "변경 전후를 같은 방식으로 확인" / "앞뒤를 같은 단계로 감싸기" |
| 종결 의식 / 마무리 의식 | "한 번에 마무리하기" / "끝낼 때 한 번에 처리" |
| 분담 통보 / 분담 명시 | "누가 뭐 할지 미리 정하기" / "역할 나누기" |
| 검증 자재 / 자재만 받기 | "확인할 재료만 받기" / "검증할 쿼리·요청만 받기" |
| 푸시백 / 가설 흔들기 | "AI 답에 한 번 더 따지기" / "그대로 받지 않고 따지기" |
| 회귀 시점 좁히기 | "언제부터 이상해졌나 짚기" |
| 객관 검토 위임 | "다른 관점으로 검토받기" |
| 사실 수집 위임 | "사실 조사 따로 시키기" |
| 떠넘기기 / 통째 위임 | "맡기기" / "한 번에 다 맡기기" |
| 시간축 진단 | "언제부터 그랬는지 시간순으로 보기" |
| 종결부 / 도입부 | "마무리 단계" / "시작 단계" |
| 본격 작업 전 컨텍스트부터 떠넘기기 | "본격적으로 일하기 전 별도 context 서브 agent에게 먼저 조사 시키기" |
| 결정성 작업 | "결정이 필요한 작업" |
| 검토 라운드 | "검토 횟수" |
| 시인성 | "잘 보이게 / 한눈에 보이게" |

**기본 원칙 2: 영어·전문 용어 풀어 쓰기**

| 금지 / 가급적 풀어 쓰기 | 풀어 쓰기 |
|---|---|
| baseline 확보 | "변경 전 데이터 확보" |
| stack trace 본문 | "에러 메시지" |
| stale (상태) | "오래된 / 캐시가 묵은" |
| 톤·범위 다듬게 | "길이·범위 다시 다듬게" |
| 본 세션 / 본 컨텍스트 | "지금 쓰는 창 / 지금 창 / 원래 창" |
| 본 컨텍스트 agent / 본 세션 agent | "지금 작업 중인 창" (지금 내가 대화 중인 메인 창. 위임받는 별도 context 서브 agent 와 구분) |
| 서브 에이전트 / 서브 agent | "별도 context 서브 agent" (지금 창 안에서 위임하는 보조 AI. "별도 창" 으로 쓰지 말 것) |
| 본인 | "나 / 내" (말투에 맞게) |
| 본인 손으로 | "내 손으로 / 직접" |
| 운영 환경 / prod | 그대로 OK (개발자 일상어) |

**기본 원칙 3: 보고서 분석 메타용어 한국어화**

분석 도구 내부 용어가 보고서 본문에 그대로 새지 않게 한다.

| 금지 (메타용어) | 풀어 쓰기 |
|---|---|
| 에피소드 / 에피소드 수 | "작업 묶음 / 주요 작업 N건" |
| 광역 모드 | "전체 기간 분석" |
| 시그니처 시퀀스 | "자주 나오는 작업 순서" |
| 미니 패턴 후보 | "작은 패턴 후보" |
| 클러스터 / 클러스터링 | "묶음 / 비슷한 것끼리 묶기" |
| 메서드러지 | "방식 / 흐름 / 습관" |
| 관용구 | "표현 / 말투" |
| 사용자 발화 / 발화 | "메시지 / 사용자가 친 메시지" |
| 사용자 첫 발화 | "첫 메시지" |
| 발화 모양 | "말투 / 메시지 형태" |
| 발화가 등장 | "이렇게 묻는 흐름이 자주 보인다" |

**기본 원칙 4: 작성 시 한 번 더 점검**

본문 한 문단을 쓰고 나면 스스로 묻는다:
- "이 단어를 신입 개발자가 한 번 읽고 바로 이해할까?"
- "이 단어를 일상 슬랙에서 쓰겠는가?"

둘 중 하나라도 NO 면 풀어 쓴다.

본문 톤: 사내 슬랙 메시지처럼 — 무게 빼고, 한자어 빼고, 한 번에 읽혀야 한다.

### outcome 시퀀스 패턴 검토 (광역 모드 필수)

`outcome_transitions` 와 `outcome_trigrams` (Stage C 가 제공) 를 보고 의미 있는 빈도 (≥5) 가 있으면 별도 패턴 후보로:

| 시퀀스 신호 | 패턴 후보 (사용자가 짚어준 형태) |
|---|---|
| `verified_by_data → verified_by_run → verified_by_data` | "테스트 데이터 추출 → 직접 실행 → 데이터 재검증" |
| `verified_by_run → verified_by_data → verified_by_run` | "실행 → 데이터 확인 → 보정 후 재실행" |
| `verified_by_data → with_changes` 빈출 | "데이터 확인 후 수정 결정" |
| `verified_by_run → committed → pushed` | "실행 검증 → 커밋 → 푸시 마무리" |
| `pushed → pr_opened → pr_with_structured_body` | "푸시 → PR 생성 → 본문 구조화" |
| `delegated_and_reported → verified_by_run → delegated_and_reported` | "위임 받아 실행 → 다시 위임" |

시퀀스 신호만으로 단정 X. 대표 에피소드의 actual turn 흐름이 일치해야 채택. 시퀀스가 빈약하면 후보 폐기.

### 예시 (나쁨)

```markdown
### 디버깅 패턴

사용자는 디버깅 작업에서 grafana mcp 와 datadog mcp 를 자주 사용한다.
(12회 관찰됨)

**왜**: 효율적이기 때문.
```

문제:
- 제목이 상황+행동 X (단순 카테고리)
- 본문이 통계 ("자주 사용") + 빈도 ("12회") 포함
- "왜" 라벨 분리됨
- 효과 설명이 추상적 ("효율적")

## 3. 도구 표기 룰

### 도구 분류

- **일반 공개 도구**: `config/public_tools_whitelist.yaml` 에 있는 도구 → 이름 유지 + 첫 등장 시 인라인 괄호
- **회사 내부 시스템** (WMS, OMS 등): 사내 공유라 그대로 노출
- **시크릿**: 항상 마스킹 (단계 7 검증에서 자동)
- **동료 이름**: 디폴트 마스킹 (단계 7 검증에서 자동)

### 인라인 괄호 표기

**규칙 1: 첫 등장 시 — "도구명(짧은 기능 설명)"**

```
"grafana mcp(로그 검색 도구)"
"datadog mcp(분산 trace 분석 도구)"
"atlassian jira(이슈 트래커)"
"slack mcp(채팅 도구)"
"confluence mcp(문서 협업 도구)"
```

**규칙 2: 두 번째 등장부터 도구명만**

```
같은 패턴 안에서 두 번째: "grafana mcp" 만
```

**규칙 3: 한 패턴 안에서만 카운트**

다른 패턴에선 다시 "첫 등장" 처리. 사용자가 패턴 단위로 읽기 때문.

**규칙 4: 기능 설명 문구는 기능 그룹 이름 기반**

`config/function_groups.yaml` 의 기능 그룹별 한국어 설명 사용:

## 4. 느슨한 그룹화 규칙

### 판정 기준

- 자연스러운 그룹이 **3개 이상** 형성되면 → 그룹화
- 그룹당 **최소 2개** 패턴
- 한 패턴이 두 그룹에 걸치면 → 그룹화 안 함 (모호)
- 그룹명은 추상적이지만 명확하게

## 5. 주별 활동 추이 (기간 ≥ 2주일 때만)

섹션 제목은 **"주별 활동 추이"** (또는 월 단위면 "월별 활동 추이"). "시계열 추이" 같은 용어는 쓰지 않는다.

### 단위 자동 결정

| 기간 | 단위 |
|---|---|
| 2-4주 | 주 단위 (W17, W18, ...) |
| 1-3개월 | 주 단위 |
| 3개월+ | 월 단위 (2026-04, 2026-05, ...) |

### 형식

```markdown
## 주별 활동 추이

### W17 (2026-04-20 ~ 2026-04-26)
주로 운영 환경 디버깅. 로그 검색 도구 사용 빈도 높음.
주요 작업 3건.

### W18 (2026-04-27 ~ 2026-05-03)
신규 기능 개발 중심으로 전환. 파일 편집 비중 늘어남.
주요 작업 5건.

### W19 (2026-05-04 ~ 2026-05-10)
확인 작업 많았음. 별도 context 서브 agent 에게 맡기는 패턴 자주 보임.
주요 작업 4건.

### W20 (2026-05-11 ~ 2026-05-17)
플러그인 절차 다듬기. 문서 작업 비중 늘어남.
주요 작업 2건.
```

각 단위 마다:
- 그 기간 주된 작업 성격 (1-2문장)
- 자주 쓴 기능 그룹 1-2개 — **한국어 설명으로** (`log_search` → "로그 검색 도구", `file_edit` → "파일 편집")
- "주요 작업 N건" 으로 표기 ("주요 에피소드" 아님)

## 6. 사용 통계 섹션

섹션 제목은 **"사용 통계"**. "사용 메타 정보" 같은 용어는 쓰지 않는다.

```markdown
## 사용 통계

**자주 쓴 Claude Code 기능 top 5**
1. 계획 모드 (Plan Mode) — 45% 작업에서 사용
2. /clear (대화 초기화) — 28%
3. 별도 context 서브 agent 호출 — 15%
4. ultrathink (깊게 생각하기) — 10%
5. 계획 모드 안에서 서브 agent 호출 — 5%

**자주 쓴 기능 묶음 top 5**
1. 로그 검색 — 32회
2. 코드 검색 — 28회
3. 파일 편집 — 24회
4. 이슈 트래커 — 18회
5. 분산 trace 분석 — 12회

**자주 쓴 도구 top 10**
1. 셸 실행 — 156회
2. 파일 읽기 — 134회
3. grafana mcp(로그 검색 도구) — 32회
...
```

라벨 어휘 규칙:
- "자주 쓰는" → "자주 쓴" (지난 기간의 분석이므로 과거형)
- "기능 그룹" → "기능 묶음"
- 도구 이름은 `config/function_groups.yaml` 의 한국어 설명을 그대로 사용 (영문 라벨 노출 X)
- Claude Code 내부 도구명(`Bash`, `Read`, `Edit` 등)은 한국어 동사로 (이미 본문에서 그렇게 쓰고 있음)
