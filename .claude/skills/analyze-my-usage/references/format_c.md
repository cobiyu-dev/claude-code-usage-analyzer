# Format C — 보고서 형식 가이드

Claude(자신)가 Stage D 작업 중 보고서 작성할 때 참조하는 형식 명세.

배경/결정 사유는 `docs/format_c_spec.md` 참고. 이 문서는 **"이렇게 작성하라"** 지시만 포함.

---

## 보고서 전체 구조

```markdown
# Claude Code 사용 패턴 분석

[메타 헤더]

---

## 주요 워크플로 패턴

[패턴 — 느슨한 그룹화 또는 평탄]

---

## 시계열 추이 (기간 ≥ 2주일 때만)

[주차별 또는 월별 요약]

---

## 사용 메타 정보

[자주 쓰는 기능/도구 top N]

---

## 참고

[raw 데이터 위치, 오류 신고 등]
```

## 2. 패턴 한 개 형식

```markdown
### [한 문장 제목 — 상황 + 행동의 핵심]

[1-3문장 본문.
 첫 문장: 상황 + 행동 시퀀스
 (선택) 중간 문장: 구체 예시 또는 변형
 마지막 문장: 효과/이유를 자연스럽게 (별도 "왜" 라벨 X)]
```

### 핵심 규칙

- 별도 필드 (왜, 변형, 예시 등) X. 본문 안에 자연스럽게 녹임
- 빈도/통계 표시 X (예: "12회 관찰" 같은 거 X)
- 도구 이름은 일반 공개 도구만 표기. 회사 내부 시스템명은 그대로 노출 가능 (사내 공유)
- 회사 내부 메서드명/엔티티명은 추상화 (`SubmitFCWorkerPackingOutbound` → `특정 워커` 등)

### 예시 (좋음)

```markdown
### 에러나 이상 신호가 보이면 관련 로그를 통째로 수집한 뒤 분석을 시작한다

운영 환경에서 에러나 특이사항이 발생하면 코드부터 보지 않고, 먼저 
**grafana mcp(로그 검색 도구)**에서 관련 access log 키를 주고 한 번에 
끌어와 전체를 훑은 뒤 본 작업에 들어간다. 가설을 좁힌 상태에서 코드를 
보기 때문에 무관한 path를 헤매지 않게 됨.

### 도구 간 식별자를 매개로 체이닝한다

한 도구의 결과에 들어있는 식별자를 추출해 다음 도구의 입력으로 넣는 
방식으로 도구를 잇는다. 예: **grafana mcp(로그 검색)**에서 dd.trace_id를 
뽑아 **datadog mcp(분산 trace 분석 도구)**로 연결. 단일 도구만으로는 
답이 안 나오는 문제를 식별자 한 줄로 깊이 분석할 수 있음.
```

### 예시 (나쁨)

```markdown
### 디버깅 패턴

사용자는 디버깅 작업에서 grafana mcp 와 datadog mcp 를 자주 사용한다.
(12회 관찰됨)

**왜**: 효율적이기 때문.
```

문제:
- 제목이 상황+행동 X (단순 카테고리)
- 본문이 통계 ("자주 사용") + 빈도 ("12회") 포함
- "왜" 라벨 분리됨
- 효과 설명이 추상적 ("효율적")

## 3. 도구 표기 룰

### 도구 분류

- **일반 공개 도구**: `config/public_tools_whitelist.yaml` 에 있는 도구 → 이름 유지 + 첫 등장 시 인라인 괄호
- **회사 내부 시스템** (WMS, OMS 등): 사내 공유라 그대로 노출
- **시크릿**: 항상 마스킹 (단계 7 검증에서 자동)
- **동료 이름**: 디폴트 마스킹 (단계 7 검증에서 자동)

### 인라인 괄호 표기

**규칙 1: 첫 등장 시 — "도구명(짧은 기능 설명)"**

```
"grafana mcp(로그 검색 도구)"
"datadog mcp(분산 trace 분석 도구)"
"atlassian jira(이슈 트래커)"
"slack mcp(채팅 도구)"
"confluence mcp(문서 협업 도구)"
```

**규칙 2: 두 번째 등장부터 도구명만**

```
같은 패턴 안에서 두 번째: "grafana mcp" 만
```

**규칙 3: 한 패턴 안에서만 카운트**

다른 패턴에선 다시 "첫 등장" 처리. 사용자가 패턴 단위로 읽기 때문.

**규칙 4: 기능 설명 문구는 기능 그룹 이름 기반**

`config/function_groups.yaml` 의 기능 그룹별 한국어 설명 사용:

```yaml
log_search       → "로그 검색 도구"
trace_view       → "분산 trace 분석 도구"
metric_query     → "메트릭 조회 도구"
db_query         → "DB 쿼리 도구"
code_search      → "코드 검색 도구"
web              → "웹 검색/조회"
issue_tracker    → "이슈 트래커"
chat             → "채팅 도구"
docs             → "문서 협업 도구"
file_read        → "파일 읽기"
file_edit        → "파일 편집"
shell_exec       → "셸 실행"
browser          → "브라우저 자동화"
design_tool      → "디자인 도구"
ide              → "IDE 연동"
agent            → "서브 에이전트"
other            → (괄호 생략 또는 도구 자체 설명)
```

## 4. 느슨한 그룹화 규칙

### 판정 기준

- 자연스러운 그룹이 **3개 이상** 형성되면 → 그룹화
- 그룹당 **최소 2개** 패턴
- 한 패턴이 두 그룹에 걸치면 → 그룹화 안 함 (모호)
- 그룹명은 추상적이지만 명확하게

### 그룹화된 출력

```markdown
## 주요 워크플로 패턴

### A. 정보 우선 (본격 작업 전 사실 데이터부터 확보)

#### 에러나 이상 신호가 보이면 관련 로그를 통째로 수집한 뒤 분석을 시작한다
[본문...]

#### 에러 디버깅 전 관련 기능의 정상 동작 설명을 먼저 만든다
[본문...]

### B. 도구 체이닝 (식별자로 도구 잇기)

#### 도구 간 식별자를 매개로 체이닝한다
[본문...]

#### 결과 식별자를 다음 작업의 진입점으로 사용한다
[본문...]
```

### 평탄 출력

```markdown
## 주요 워크플로 패턴

### 에러나 이상 신호가 보이면 관련 로그를 통째로 수집한 뒤 분석을 시작한다
[본문...]

### 도구 간 식별자를 매개로 체이닝한다
[본문...]

### 복잡한 검토는 별도 에이전트에 위임한다
[본문...]
```

## 5. 시계열 섹션 (기간 ≥ 2주일 때만)

### 단위 자동 결정

| 기간 | 단위 |
|---|---|
| 2-4주 | 주 단위 (W17, W18, ...) |
| 1-3개월 | 주 단위 |
| 3개월+ | 월 단위 (2026-04, 2026-05, ...) |

### 형식

```markdown
## 시계열 추이

### W17 (2026-04-20 ~ 2026-04-26)
주로 프로덕션 디버깅. log_search 기능 빈도 높음.
주요 에피소드 3개.

### W18 (2026-04-27 ~ 2026-05-03)
신규 기능 개발 중심으로 전환. file_edit 비중 증가.
주요 에피소드 5개.

### W19 (2026-05-04 ~ 2026-05-10)
검증 작업 다수. agent 위임 패턴 빈도 높음.
주요 에피소드 4개.

### W20 (2026-05-11 ~ 2026-05-17)
플러그인 절차 다듬기. docs 비중 증가.
주요 에피소드 2개.
```

각 단위 마다:
- 그 기간 주된 작업 성격 (1-2문장)
- 자주 쓴 기능 그룹 1-2개
- 주요 에피소드 수

## 작성 후 자가 점검 (보고서 완성 직전)

다음 항목을 마지막으로 한 번씩 점검:

**도구 표기**
- [ ] 모든 일반 공개 도구의 첫 등장에 인라인 괄호 있나?
- [ ] 두 번째 등장부터 도구명만 쓰여있나?
- [ ] 한 패턴 안에서만 카운트했나?

**패턴 형식**
- [ ] 모든 제목이 "상황 + 행동" 한 문장인가?
- [ ] `**왜**:` 같은 별도 라벨 없나?
- [ ] 빈도/통계 표시 없나? (메타 정보 섹션 제외)

**도메인 무관성**
- [ ] 패턴이 다른 직군 사람도 자기 도구로 치환해서 읽을 수 있나?
- [ ] 회사 내부 메서드명/엔티티명 그대로 노출되지 않았나? (회사 시스템명은 OK)

**누락**
- [ ] 메타 헤더 빠진 필드 없나?
- [ ] 기간 ≥ 2주면 시계열 섹션 있나?
- [ ] 참고 섹션 있나?

자가 점검 후 임시 위치에 저장. 단계 7 (검증) 으로 넘어감.
